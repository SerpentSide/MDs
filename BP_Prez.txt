Dobrý den, jmenuji se Jan Procházka a dneska vám představím svou bakalářskou práci na téma ‚GUI pro aplikaci analyzující schéma databáze a SQL dotazy‘, kterou jsem vypracoval pod vedením pana docenta Radima Bači.
Nejprve stručně představím cíl práce, následně použité technologie metodiku a poté hlavní výsledky.

Na katedře informatiky je vyvíjen nástroj zvaný DBShepherd umožňující analyzovat databázová schémata a SQL dotazy z hlediska redundance a výskytu databázových antipatternů.
Původní řešení fungovalo pouze jako command-line aplikace, což komplikovalo práci zejména méně technicky zaměřeným uživatelům a zhoršovalo interpretaci výsledků.
Motivací této práce proto bylo vytvořit webovou aplikaci, která práci s nástrojem zjednoduší a zároveň doplní výsledky analýzy o vysvětlení a doporučení generovaná chatbotem.


Vypracovaná část je implementována na principu mikroservisní architektury. První službou je backend v Pythonu. Ten komunikuje s původním systémem prostřednictvím jeho REST API. Zároveň přistupuje k pracovní databázi a externímu LLM API, které může zadat uživatel. Backend také poskytuje vlastní REST API, které využívá frontendová aplikace implementovaná v JavaScriptu. Druhou službou je tedy frontend, který zajišťuje uživatelské rozhraní a komunikaci s backendem.

Na levém obrázku lze vidět strukturu celého projektu, pro účely této obhajoby ořezaného o části, které nejsou přímo potřebné pro spuštění celé implementace. Tato bakalářská práce přidala složky gui (frontend), userApi (backend) a BAKALRSKA_PRACE_README, popisující podrobněji autorův přínos. V každé složce je dodatečný README soubor, popisující kroky spuštění dané služby včetně instalací potřebných knihoven, nebo vytvoření vlastního .env souboru podle .env.example.

Backend byl realizován s využitím frameworku FastAPI, který zajišťuje právě zmíněnou komunikaci s frontendem. Tato
služba plní roli prostředníka mezi analytickou částí, frontendem a externím llm api. Zároveň komunikuje s databázi a stará se správu uživatelů, skupin a api klíčů.
Hesla jsou ukládaná pomocí knihovny bcrypt, využívající stejnojmennou hashovací funkci.
Identita přihlášeného uživatele se kontroluje s využitím JWT tokenů pomocí dependency injection mechanismu frameworku FastAPI, konkrétně prostřednictvím třídy OAuth2PasswordBearer.
Pro volání LLM odpovědi byla využita knihovna litellm, která sjednocuje volání různých modelů a providerů.

Frontendová část aplikace byla implementována v jazyce JavaScript s využitím knihovny React, která umožňuje tvorbu komponentově orientovaného uživatelského rozhraní. Pro vývojové prostředí a build proces byl použit nástroj Vite, který zajišťuje rychlé spouštění aplikace. Uživatelské rozhraní bylo vytvořeno pomocí komponent knihovny shadcn/ui, která poskytuje moderní a znovupoužitelné UI komponenty postavené nad Tailwind CSS.

Na této stránce jde vidět hlavní podstata celé bakalářské práce. Jedná se o stránku DatabaseDetail a většina funkcionality se odehrává právě zde. Začnu-li ze stran, na levé straně obrazovky je dashboard s navigací na základní body webové stránky, který je přítomný po většinu práce. Horní bar ukazuje název právě sledované databáze 'Test DB'.
Ve středu stránky jsou pod sebou vypsané všechny sql dotazy používaného schéma (v rámci ukázky jenom jeden dotaz, aby se vše vešlo na stránku). Textbox na přidání dalšího sql dotazu do schématu a textbox na vytvoření nového schématu. V horní pravé části je tlačítko Analyze, které provede analýzu všech sql dotazů a vypíše analyzované chyby v červeném listu.
Vedle tlačítka analyze je zleva rolldown všech LLM daného uživatele, a právě vybraný se použije pro přepis pomocí LLM v případě, že uživatel klikne na tlačítko Rewrite, v pravém horním rohu každého sql dotazu. Popřípadě může uživatel označit checkboxem v levém horním rohu více dotazu a zmáčknutím Rewrite selected přepíše všechny označené. Přepsané sql, popřípadě chyba v přepisu se zobrazí v modrém poli s možností daný dotaz zkopírovat jedním tlačítkem.

Na tomto slidu jde vidět část sekvenčního digramu, pro účely této prezentace oříznutý na analýzu a přepis sql dotazu. Přeskočil jsem přihlašování a načítaní databáze schémata a dotazů. Frontend volá pouze backend a ten orchestruje veškerou komunikaci, frontend nemá o DBShepherd povědomí, takže je možné měnit a upravovat jednotlivé služby nezávisle na sobě. Když frontend zavolá analýzu, pošle id databáze a v headers svůj jwt token. Backend si obstará poslední schéma, analyzuje všechny dotazy a v případě úspěchu i všechny analyzované problémy pro každý sql dotaz. Poté vrátí frontendu pouze výsledné chyby pro dané sql dotazy. Přepis funguje podobným způsobem. Frontend poue zavilá endpoint s id sql dotazu a id llm api které chce použít. Backend jako první zkontroluje limit, a v případě překročení vrátí chybu o limitu ještě před tím, než začne posílat požadavky na DBShepherd. Backend si sežene potřebné informace o sql jako například chyby, tabulky, které se využívají a jejich atributy. Tyto informace vloží do předpřipraveného templatu s promptem a zavolá llm dle llm api id. Výsledek v json formátu posléze vrátí frontendu, který si zkontroluje, zda se přepis povedl či nikoli podle atributu status.
